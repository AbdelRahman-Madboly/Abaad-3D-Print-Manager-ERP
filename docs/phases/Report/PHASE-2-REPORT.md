## Wizard approach

`SetupWizard` keeps a single `_card_frame` container. `_show_step(n)` destroys
all children of the card frame and calls `_build_step_N()` to rebuild it.
`_refresh_buttons(n)` re-packs Back/Skip/Close/Next-or-Finish in the correct
order each time (pack_forget all → re-pack in declared order).

Per-step data lives in preserved `StringVar` dicts (`_vars`, `_filament_vars`,
`_printer_vars`, `_review_vars`): on first build a key is missing so a new
StringVar is created with the default; on subsequent builds the existing var
is reused, so user edits survive Back/Next navigation.

Double-commit is prevented by `_filament_committed` and `_printer_committed`
flags that block re-running `_commit_filament()` / `_commit_printer()` if
the user presses Back from Step 3/4 and Next again.

Step 4's "Finish Setup" button calls `_commit_all_settings()` which saves
company fields, currency, rate, cost defaults, and `setup_complete = "1"` in
one `db.save_all_settings()` batch call.
