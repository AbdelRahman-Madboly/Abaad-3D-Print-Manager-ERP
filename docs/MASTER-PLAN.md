## Product goal

Turn a single-business hardcoded app into a clean, generic, brandable
desktop ERP that any 3D-printing service business can deploy fresh,
configure through a first-run wizard, and use daily — on Ubuntu or Windows —
without touching source code.

**Not** a SaaS. Not a web app. A well-built desktop app with a clean
architecture that could grow into those things later if needed.

---

## The two-identity model

```
PRODUCT  = Abaad (the maker's app identity) — fixed in config.py
TENANT   = the business owner's company     — configured at first run, lives in DB
```

A generated invoice: tenant logo + name + contact at **top**,
`"Generated by Abaad ERP"` small credit at **bottom**.

## Phase sequence and rationale

| # | Phase | Rationale |
|---|-------|-----------|
| 0 | Repo audit & honest baseline | Know exactly what's there before changing anything |
| 1 | Core stabilization | Get the foundation solid: DB layer, settings, all tests green |
| 2 | Tenant brand system | Core product promise: any business can use this |
| 3 | Dashboard & analytics verify | Confirm Phase 4 work is real and tests are meaningful |
| 4 | Git workflow & CI | Professional foundation before more feature work |
| 5 | Launchers (Ubuntu + Windows) | Unblocks real daily use now |
| 6 | Cross-platform polish | Fonts, icons, consistent look on both OSes |
| 7 | PDF service | Tenant body + Abaad footer — completes the branding model |
| 8 | Documentation | Architecture, user guide, developer guide |
| 9 | Packaging (PyInstaller) | Single-file distribution for non-technical users |

**Why this order:** baseline → branding → CI → usability → distribution.
CI comes before more features so nothing regresses silently.
Launchers come before packaging because they unblock daily use immediately.

## Architecture rules (repeated here for the architect)

- Tabs call services only — no business logic in UI
- Services contain all business rules
- DatabaseManager returns plain `dict`s — no models at the DB layer
- `config.py` = product constants + fallback defaults only
- `settings` table = all user-configurable values
- No generic SQL methods on DatabaseManager
- Every change keeps `pytest` green

## DB abstraction for future Postgres migration

All SQL in DatabaseManager typed methods. Services never touch SQL.
A future `PostgresDatabaseManager` reimplements the same interface.
This is the "SQLite now, Postgres later" design.
